## 2. AI Filtering & Formatting Rules ("Rule Nhập Việc")

When summarizing daily Rocket.Chat logs, you must strictly follow these rules:

### A. Focus & Relevance Filtering
- **Focus Areas**: Only extract conversations directly involving `Chu Văn Mai` regarding:
  - Task assignments, resource allocation, and team coordination.
  - Technical troubleshooting (e.g. App Store Connect, iOS Subscription configurations).
  - Business discussions (e.g. copyright registration, monetization).
  - Performance reviews and HR decisions (e.g. intern probation evaluations).
  - **Meeting Announcements & Petitions**: Scan notification channels (see Section F below) to capture calendars, meetings, and system approvals.
- **Noise Elimination**: Automatically filter out and discard:
  - General chat, gossiping, and jokes.
  - Team logistics (e.g. company trip details, vehicle arrangements).
  - Food/drink orders (e.g. milk tea, noodles, lunch plans).
  - Broadcasted messages where the user did not actively participate.

### B. Title / Summary Formatting Rule (CRITICAL)
The title/summary of each task must follow this exact formula and length requirement:
- **Formula**: `[Làm gì / Hành động cụ thể] - [Để làm gì / Mục tiêu cụ thể]`
- **Length**: **Must be at least 50 characters long** (tối thiểu 50 ký tự). Expand the description of the action and objective if necessary to meet this requirement.
- *Good Example*: `Khắc phục lỗi lag ảnh sản phẩm IAP và cấu hình lại mã ID gói sản phẩm trên iOS - Để đảm bảo thông tin gói IAP hiển thị đầy đủ và chính xác trên App Store.` (166 characters)

### E. Project Association Rule (CRITICAL)
Every task must be mapped to the correct project:
- **Strict Rule**: **DO NOT** make assumptions, propose, guess, or suggest any project name yourself (even if the task comes from a group chat with a recognizable project name).
- **Action**: You **MUST** always list all extracted work tasks, specify their chat source room, and **explicitly ask the user to reply with the exact project name for each individual task**.
- Only write to `tasks.json` or trigger the submission *after* the user has provided the exact project mappings.

### F. Notification & Meeting Channel Scanning Rule (NEW)
As a Team Lead and PM, many of the user's critical activities (meetings, approvals, and coordinate tasks) are recorded in notification channels:
- **Rule**: You **must scan** all chat rooms containing the words **"Notification"** or **"Petition"** (e.g. `Horus Notification`, `PMNotification`).
- **Meeting Logs**: Look for calendar invite messages (e.g. *Phạm Tiến Doanh đã tạo lịch có bạn... Sự kiện: Họp nội bộ*) to automatically extract meeting attendances as work tasks.
- **Approval Logs**: Look for personnel petition notifications (e.g. *Nhân sự đã tạo đơn yêu cầu: Nghỉ phép, Đi muộn*) to log PM/administrative tasks (reviewing and approving team petitions).

### G. Direct Message (DM) & Private Chat Scanning Rule (NEW & CRITICAL)
To ensure no direct team communications or PM assignments are missed:
- **Rule**: You **MUST** scan all private direct message (DM) channels and private group chats that had active exchanges during the day.
- **Technical Sync Implementation**: The local sync engine (`sync_rocket.py`) must evaluate the Last Message timestamp (`lm` or `_updatedAt`) of all 300+ subscriptions against the start of today to identify **every active DM with specific individuals** (e.g. `longvv`, `nhihy`, `quynhpt`) and fetch their message history automatically.

### H. Channel Blacklisting / Exclusion Rule (NEW & CRITICAL)
To prevent cluttering the sync logs with social and logistics chat rooms:
- **Exclusion List**: The following default channels/groups must be skipped and filtered out entirely from the sync: `HorusXamF`, `ComNuoc`, `CoThucMoiVucDcDao`, `CungNhauGiamCan`.
- **User Customization**: The user can easily customize and add more excluded rooms inside the local `.env` file under the key `ROCKET_EXCLUDED_ROOMS` (comma-separated).
- **Behavior**: The sync script `sync_rocket.py` will read this key and dynamically discard any matching channel (case-insensitive) during the sync phase.

### I. Task Completion Status Rule (CRITICAL)
Since the logged tasks represent work that has already been successfully executed and completed during the day:
- **Rule**: All logged tasks must be marked as **Done / Hoàn thành** directly on the WorkAI portal.
- **Data Representation**: Every task object stored in `tasks.json` must contain the key `"status": "Done"`.
- **Automation Behavior**: The `submitter.py` script must automatically transition each created issue to the **Done / Hoàn thành** state by opening the issue sheet and selecting the Done status.

### J. Active Sprint Mapping Rule (NEW & CRITICAL)
To ensure tasks are recorded under the active project lifecycle:
- **Rule**: All logged tasks must be assigned to the **newest / active Sprint** (Sprint mới nhất) for their respective projects on WorkAI.
- **Data Representation**: Every task object stored in `tasks.json` must contain the key `"sprint": "latest"`.
- **Automation Behavior**: The `submitter.py` script must read this property and automatically select the latest active Sprint dropdown option during submission.

### K. Work Date Setting Rule (NEW & CRITICAL)
To ensure all entries are recorded under the correct working calendar date:
- **Rule**: All logged tasks must be assigned and logged under the **exact date they were performed** (defaulting to the current local calendar date `YYYY-MM-DD` of the sync run).
- **Data Representation**: Every task object stored in `tasks.json` must contain the key `"date": "YYYY-MM-DD"`.
- **Automation Behavior**: The `submitter.py` script must read this property and ensure the date picker/input field on the WorkAI portal is selected and matched to this exact date during submission.

### L. Daily Time Allocation Rule
WorkAI utilizes a Daily Time Allocation (timesheet calendar grid) to track timesheets:
- **Rule**: All logged tasks must be placed into the correct day column of the **Daily Time Allocation** calendar. Adjusting the allocated hours (duration) will be managed **manually by the user**.
- **Automation Behavior**: The `submitter.py` script creates issues via the "**+ Thêm issue**" button on the correct day column, which automatically places the issue into that day's schedule. The script does **NOT** adjust hours (default 0.1h) — the user will manually change the time allocation afterwards.

### `/tonghop` — Quét chat & Lên danh sách đầu việc

**Phạm vi**: Thu thập dữ liệu → Lọc → Đối chiếu file lưu trạng thái → Liệt kê → Hỏi project mapping → Lưu trạng thái.

**Các bước thực hiện (CHỈ những bước này):**
1. Chạy `python sync_rocket.py` để đồng bộ Rocket.Chat → `chat_raw.json`
2. Đọc `chat_raw.json` và áp dụng Rules A–H (Section 2) để lọc noise, trích xuất đầu việc.
3. Đọc file `approved_tasks.json` (nếu có) để đối chiếu:
   - Loại bỏ các đầu việc user đã yêu cầu xóa trước đó.
   - Giữ nguyên các đầu việc user đã gán project.
   - Chỉ thêm vào các đầu việc MỚI phát sinh.
4. Hiển thị danh sách đầu việc được cập nhật cho user (kèm source room).
5. Hỏi user kiểm tra: loại bỏ các việc không cần thiết và gán project cho việc mới.
6. Sau khi user trả lời → Lưu lại trạng thái danh sách cuối cùng (bao gồm các việc đã xóa và các việc đã gán project) vào `approved_tasks.json`.
7. Phân bổ giờ (tổng = 8.0h) cho các việc hợp lệ.
8. **DỪNG LẠI** — Không viết `memorytask.md`, không tạo nội dung chi tiết.

**Output**: Danh sách đầu việc hiển thị trong chat và được lưu trạng thái vào `approved_tasks.json`.

### `/taoviec` — Viết nội dung chi tiết & Lưu memorytask.md

**Phạm vi**: Lấy danh sách đầu việc từ file lưu trạng thái → Viết nội dung → Lưu file.

**Điều kiện**: `/tonghop` đã chạy xong, user đã duyệt và file `approved_tasks.json` đã được cập nhật.

**Các bước thực hiện (CHỈ những bước này):**
1. Đọc danh sách đầu việc hợp lệ từ file `approved_tasks.json`
2. Với mỗi đầu việc, viết nội dung theo Rules B:
   - **Title**: Theo formula `[Hành động] - [Mục tiêu]` (≥50 ký tự)
3. Lưu toàn bộ vào file `memorytask.md` với format chuẩn (xem bên dưới)
4. **DỪNG LẠI** — Không chạy submitter.py, không tạo tasks.json

**Output**: File `memorytask.md` trong workspace.

**Format `memorytask.md`:**
```markdown
# Daily Tasks — YYYY-MM-DD

## Task 1
- **Project**: [tên project]
- **Duration**: [X.Xh]
- **Title**: [tiêu đề ≥50 ký tự]

...
```

### `/nhapviec` — Nhập task vào WorkAI

**Phạm vi**: Đọc `memorytask.md` → Tạo `tasks.json` → Chạy `submitter.py`.

**Điều kiện**: File `memorytask.md` đã tồn tại và có nội dung.

**Các bước thực hiện (CHỈ những bước này):**

7. **DỪNG LẠI** — Báo kết quả, nhắc user tự chỉnh giờ phân bổ

**Output**: Issues được tạo trên WorkAI portal.
